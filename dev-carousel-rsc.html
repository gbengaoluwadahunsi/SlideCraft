Definition

What Are Server Components?

Components that run exclusively on the server. They render to HTML, never ship JavaScript to the client, and can directly access databases, file systems, and secrets.

// This never reaches the browser
async function UserProfile({ id }) {
  const user = await db.query(`SELECT * FROM users WHERE id = ${id}`);
  return <h1>{user.name}</h1>;
}

The Problem

Why Do We Need Them?

Traditional React ships everything to the browser — components, data-fetching logic, dependencies. The result:

  • Massive JS bundles slowing first paint
  • Waterfalls: render → fetch → render → fetch
  • Client secrets leaking into bundles
  • Hydration cost on every page load
  • Duplicate logic on server + client

Mental Model

Think in Layers

Server components are the outer shell. Client components are interactive islands inside them. Data flows down, interactivity is opt-in.

// Server Component (default)
export default async function Page() {
  const data = await getData();
  return (
    <Layout>
      <InteractiveWidget data={data} />
    </Layout>
  );
}

Data Fetching

async/await in Components

No useEffect. No loading states to manage. No client-side fetch waterfalls. Just write async functions.

async function ProductList() {
  const products = await db.product.findMany();
  return products.map(p =>
    <Card key={p.id} product={p} />
  );
}

Streaming

Progressive Rendering

Suspense boundaries let slow parts stream in later. The shell loads instantly while data-heavy components resolve.

<Suspense fallback={<Skeleton />}>
  <SlowDataComponent />
</Suspense>

// Shell renders immediately
// SlowDataComponent streams in when ready

Server Actions

Functions marked with "use server" run on the server but can be called from client components. No API routes needed.

Watch Out

Common Pitfalls

  • Accidentally importing a client lib in a server component
  • Passing non-serializable props across the boundary
  • Over-using "use client" — making everything a client component
  • Forgetting Suspense boundaries for async components
  • Trying to use hooks (useState, useEffect) in server components
  • Not understanding when revalidation happens

What's Next

The Future

RSCs are just the beginning. The React team is building toward a world where the framework decides what runs where — automatically.

  • Partial prerendering (PPR) — static shells + dynamic holes
  • Server components in React Native
  • Cross-framework RSC protocol
  • Compiler-driven client/server splitting
  • Edge-first by default
